# PM_agent 项目完整总结文档

## 一、项目概述

**PM_agent**（面试模拟Agent系统）是一个面向AI产品经理面试场景的模拟系统。系统核心角色设定为：
- **用户（面试官）**：负责向Agent提问
- **Agent（面试者）**：基于简历和知识库回答问题

**核心价值**：帮助AI产品经理准备面试，通过模拟真实面试场景提升应答能力。

## 五、主要技术详解

### 5.1 混合检索技术（核心创新）

#### 5.1.1 技术选型理由

| 技术 | 解决的问题 | 优势 |
|------|-----------|------|
| 向量检索 | 语义理解 | 容错性强，理解同义词 |
| BM25 | 精确匹配 | 关键词命中率高 |
| RRF融合 | 结果排序 | 不依赖分数归一化，对排名敏感 |

#### 5.1.2 RRF算法详解

**公式：**
```
RRF_score(d) = Σ 1/(k + rank_i(d))

其中：
- k = 60（平滑常数）
- rank_i(d) = 文档d在第i个结果列表中的排名
```

**示例：**

| 文档 | 向量排名 | BM25排名 | RRF计算 | 最终分数 |
|------|---------|---------|---------|---------|
| doc1 | #1 | #1 | 1/61 + 1/61 | **0.0328** |
| doc2 | #2 | 无 | 1/62 + 0 | 0.0161 |
| doc3 | 无 | #2 | 0 + 1/62 | 0.0161 |

**关键洞察：**
- 双来源确认的文档分数翻倍
- 排名比绝对分数更重要
- 新文档（只在一个列表中）也能参与排序

#### 5.1.3 中文分词优化

```python
# 使用jieba进行中文分词
import jieba

def _tokenize(self, text: str) -> List[str]:
    # 检测中文
    if re.search(r'[\u4e00-\u9fff]', text):
        # 使用jieba.cut_for_search模式
        words = list(jieba.cut_for_search(text))
    else:
        words = text.split()
    return words
```

## 六、核心解决方案

### 6.1 混合检索方案

**解决的问题：**
1. 纯向量检索对精确关键词匹配不足
2. 纯BM25缺乏语义理解能力
3. 单一检索方式召回率低

**方案架构：**

```
┌─────────────────────────────────────────┐
│           混合检索架构                   │
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────────┐    ┌─────────────┐    │
│  │  向量检索    │    │  BM25检索   │    │
│  │  (语义)     │    │  (关键词)   │    │
│  │             │    │             │    │
│  │ • 余弦相似度 │    │ • jieba分词 │    │
│  │ • 768维向量 │    │ • SQLite FTS│    │
│  │ • ChromaDB  │    │ • 倒排索引  │    │
│  └──────┬──────┘    └──────┬──────┘    │
│         │                   │           │
│         └─────────┬─────────┘           │
│                   ▼                     │
│            ┌─────────────┐              │
│            │  RRF融合    │              │
│            │             │              │
│            │ • k=60     │              │
│            │ • 倒数排序 │              │
│            │ • 双源加分 │              │
│            └──────┬──────┘              │
│                   ▼                     │
│            ┌─────────────┐              │
│            │  TopK结果   │              │
│            │  +来源标记  │              │
│            └─────────────┘              │
│                                         │
└─────────────────────────────────────────┘
```

**效果提升：**

| 查询 | 纯向量 | 混合检索 | 提升 |
|------|--------|----------|------|
| "什么是RAG" | 2条 | 3条 | +50% |
| "向量数据库" | 0条 | 3条 | 从0到3 |
| "BAAI" | 0条 | 3条(双源) | 精确匹配 |

### 6.3 上下文压缩方案

**问题：** 长对话历史导致Token消耗过大

**解决方案：**

1. **对话摘要**：每轮对话生成结构化摘要
   ```python
   {
     "round_number": 1,
     "question": "问题摘要",
     "summary": "回答摘要（200字）"
   }
   ```

2. **分层存储**：
   - `messages`：完整消息（用于显示）
   - `conversation_history`：摘要列表（用于Prompt）

3. **截断策略**：只保留最近5轮摘要
